# ADR-0009 学生作业 · 推演记录回放

- 状态：已采纳
- 日期：2026-05-03
- 版本：v1.9.0

## 背景

v1.8.0 已支持老师创建课堂任务、学生在沙盘里完成「验收任务」，但老师只能拿到学生口头反馈或截图，无法横向对比全班——例如：

- 谁卡在哪里没抵达终局？
- 是否大多数学生都误入某条岔路？
- 哪个合格终局最受欢迎，哪个被忽视？

需要一份「按 task_id 聚合所有学生 playthrough」的回放视图。

## 决策

### 数据模型变更

- `PlaythroughSnapshot` 增字段 `task_id: str | None`、`student_name: str | None`
- 字段默认 `None`，对自由探索模式的旧记录完全兼容
- DB 镜像表 `sandbox_playthroughs` 仍走整段 JSON 覆盖，无需 schema 迁移

### 服务层

- `engine.list_playthroughs_by_task(task_id)`：纯内存过滤
- 不引入冗余索引；当前数据规模（单课至多几十名学生）线性扫描足够

### Router 层

- `POST /playthroughs?scenario_id=&task_id=&student_name=` 三参数全可选
- `GET /classroom/tasks/{id}/submissions` 返回 `SubmissionsAggregate`：
  - `node_visit_counts` / `edge_traverse_counts` / `terminal_distribution`：对所有学生的 history 做计数
  - 每位学生一条 `SubmissionItem`，复用 `_check_one` 抽离出的验收算法
- 不分页：合理预期单课提交 < 100 条；前端 Table 自带分页足矣

### 前端

- SandboxPage：任务模式下学生姓名为必填，sessionStorage 缓存避免重复
- ClassroomPage：每张任务卡加「作业回放」按钮，Drawer 内含统计三宫格 + 终局分布 + 节点/边 Top 10 热力 + 学生提交 Table（行展开看完整 history）

## 替代方案

- **持久化每条 SubmissionItem 到独立 submissions 表**：拒绝。验收算法稳定且廉价，存原始 playthrough 即可重算；避免双源冗余
- **WebSocket 推送实时回放**：拒绝。课堂场景老师手动刷新足够，过度设计
- **强制学生登录**：拒绝。当前为研究原型，匿名+学生名足以追溯

## 风险

- 学生重名可能造成混淆 → 后续可考虑「班级 · 学号」或加 student_id
- 学生若反复点「开始新推演」会留下多份提交 → Table 默认按 updated_at 倒序，老师可看到最新一条

## 验证

```
for name in 小明,小红,小李:
  POST /playthroughs?task_id=task_xxx&student_name=$name
  → 沿推荐路径走完
GET /classroom/tasks/task_xxx/submissions
→ {
    total_count: 3,
    accepted_count: 3,
    node_visit_counts: { n_call:3, n_reform:3, n_field_survey:3, n_pacify:3, n_dynasty:3 },
    edge_traverse_counts: { e_call_reform:3, ... },
    terminal_distribution: { n_dynasty: 3 }
  }
```
