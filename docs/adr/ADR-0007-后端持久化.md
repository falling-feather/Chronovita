# ADR-0007 后端持久化（SQLAlchemy + SQLite）

- 状态：已采纳
- 日期：2026-05-03（v1）/ v0.1.0 重启时复核
- 版本：v0.1.0

## 背景
平台需要一个零配置、可在课堂笔记本电脑离线运行的持久化方案。开发期不引入 Postgres、Redis；生产期再迁移。

## 决策

### 1. 选型
- 复用 `sqlalchemy==2.0.35`（同步 Engine + Core API），**不**引入 ORM Session、不引入 Alembic、不引入 asyncpg
- 后端使用 SQLite 本地文件（`data/chronovita.db`），零外部服务依赖
- 表设计采用「文档式 JSON 镜像」：每张业务表只有 `主键 / data TEXT / updated_at`，序列化整段 Pydantic 模型，避免与各业务多变的字段表结构耦合

### 2. v0.1.0 当前状态
- `services/persistence/` 仅保留通用 `init_engine` + 通用 `kv` 工具表骨架
- 课程业务（课程、章节、学习进度等）的具体表结构在阶段 1 落地时按需追加
- `apps/api/main.py` lifespan 调用 `init_engine` 创建空库

### 3. 命名约定（后续业务表追加时遵循）
- 表名使用模块名复数：`courses` / `learning_records` / `practice_attempts` 等
- 主键统一 `xxx_id: TEXT`
- 数据列统一 `data: TEXT (JSON)`
- 时间戳列统一 `updated_at: DATETIME`

## 后果

正面：
- 单文件持久化，方便老师/学生自带电脑跑课
- JSON 镜像表对模型变更友好，无需 Alembic 迁移即可演进 Pydantic 模型

负面：
- SQLite 并发写入弱；超过 50 并发时需切换到 Postgres
- JSON 列无法直接走 SQL JOIN；查询需在内存里组装
