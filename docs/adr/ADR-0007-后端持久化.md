# ADR-0007 后端持久化（SQLAlchemy + SQLite）

- 状态：已采纳
- 日期：2026-05-03
- 版本：v1.6.0

## 背景
v1.0–v1.5 的 sandbox / agent / canvas 三模块全部依赖进程内字典（`_PLAYTHROUGHS` / `_SESSIONS` / `_BOARDS`）。
`uvicorn --reload` 或重启即丢失全部学习数据，无法支持「跨设备 / 多会话 / 课堂复用」场景。
本期需在不打断现有同步 router 调用风格、不引入新依赖的前提下，让数据落盘并在启动时自动恢复。

## 决策

### 1. 选型
- 复用已声明的 `sqlalchemy==2.0.35`（同步 Engine + Core API），**不**引入 ORM Session、不引入 alembic、不引入 asyncpg
- 后端使用 SQLite 本地文件（`data/chronovita.db`），零外部服务依赖
- 表设计采用「文档式 JSON 镜像」：每张表只有 `主键 / data TEXT / updated_at`，序列化整段 Pydantic 模型，避免与各模块多变的字段表结构耦合

### 2. 模块拆分
- 新增 `services/persistence/`
  - `db.py`：声明三张表（`canvas_boards` / `sandbox_playthroughs` / `agent_sessions`）+ `init_engine` + `_upsert / _delete / _load_all` 私有助手 + 三组对外 `save_* / delete_* / load_all_*`
  - `__init__.py`：聚合公共 API
- `apps/api/settings.py`：新增 `sqlite_path: str = "../../data/chronovita.db"`（uvicorn cwd 在 `apps/api/`，相对回到仓库根）
- `apps/api/main.py` lifespan：`init_engine` → `canvas_store.hydrate_from_db()` → `sandbox_engine.hydrate_from_db()` → `agent_dialogue.hydrate_from_db()`

### 3. 接入策略：内存为主，DB 为镜像
- 写入路径：`create_board / upsert_node / upsert_edge / delete_node / delete_edge / delete_board / new_playthrough / advance / new_session / stream_answer 末尾` 全部追加 `persistence.save_*` 或 `persistence.delete_*`
- 读取路径不变：仍走内存字典，热查询零额外 IO
- 启动路径：lifespan 装载所有持久化记录回内存

### 4. 序列化协议
- `_dumps` 用 `json.dumps(ensure_ascii=False, default=_json_default)`
- `_json_default`：`datetime → isoformat()`；`hasattr(o, "model_dump") → o.model_dump(mode="json")`
- 反序列化用 `Model.model_validate(raw)`，失败的脏数据 silently skip 不阻塞启动

## 后果
- 优点：进程崩溃 / `--reload` / Windows 重启后，全部学习数据可恢复；前端零改动；router 零改动；无新依赖
- 限制：写入是「整段 JSON 覆盖」而非字段级，单板节点/边非常多时（~1k+）每次 save 会有冗余 IO；后续 v1.6.x 可拆细
- 限制：未做并发锁，假设 uvicorn 单 worker

## 验证
1. 启动 → 调用 `POST /api/v1/canvas/boards` 创建测试板 → SQLite 中 `canvas_boards` 表出现该 row
2. 触发 lifespan reload（修改 main.py timestamp）→ `GET /api/v1/canvas/boards` 仍返回该板（hydrate 成功）
3. `DELETE /api/v1/canvas/boards/<id>` → DB 行消失
4. 三表 `count(*)` 与内存 list 长度一致
